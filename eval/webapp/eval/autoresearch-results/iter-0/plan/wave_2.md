# Wave 2 — Full-Text Search and Cursor Pagination

### 2.1 Search migration
- intent: Add a migration that: (a) adds a `search_vector TSVECTOR` column to the `posts` table, (b) creates a GIN index on it, (c) creates a trigger function that automatically updates `search_vector` from `title` and `body` using `to_tsvector('english', title || ' ' || body)` on INSERT and UPDATE, (d) backfills existing rows.

### 2.2 Search endpoint
- intent: Add `GET /api/posts/search?q=<query>` endpoint to the posts router. Uses `to_tsquery('english', <query>)` to search against the `search_vector` column. Returns matching posts ordered by relevance (`ts_rank`). Each result includes a `headline` field generated by `ts_headline('english', body, query)` that highlights matching terms. Returns empty array for no matches. Returns 400 if `q` parameter is missing.
- files:
    - src/routes/posts.ts (MODIFY)
- tests:
    - test/search.test.ts
- verify: bun test test/search.test.ts

### 2.3 Cursor-based pagination
- intent: Add cursor-based pagination to `GET /api/posts` and `GET /api/users`. Accept optional query params `?limit=N&cursor=<opaque>`. The cursor is a base64url-encoded JSON `{id: number, created_at: string}`. Response format: `{data: [...], next_cursor: "..." | null, has_more: boolean}`. Default limit is 10, max limit is 100. When no cursor is provided, start from the most recent. The existing behavior (return bare array) must change to the new envelope format. IMPORTANT: Also update `test/users.test.ts` and `test/posts.test.ts` to handle the new envelope format — any assertion that expects a bare array from GET /api/users or GET /api/posts must unwrap `response.body.data` instead. Run `bun test` (all tests) to verify nothing is broken.
